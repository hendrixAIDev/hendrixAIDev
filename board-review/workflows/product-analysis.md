# Product Analysis

**Purpose:** Guide CTO planning when the next useful step is analysis rather than implementation.

**What this covers:**
- When to use the product analysis workflow
- Planning expectations before dispatch
- Typical analysis, decision, follow-up, and closure path

Use when the right next step is analysis, not implementation.

Default path:
1. CTO triage and planning
2. Product analysis dispatch
3. CTO decision
4. Create follow-up tickets, dispatch another workflow, escalate, or close

CTO planning should define:
- the decision the analysis must unblock
- source docs, user evidence, or product context to inspect
- expected output format
- whether output should update the task queue, product docs, or GitHub issue only

Notes:
- PM/product-discovery passes should update the project-local task queue by default
- CTO promotes queue items into GitHub tickets during board-review queue intake
- direct ticket creation is allowed only when CTO explicitly requests it
- no engineering work is required by default
- CTO decides what happens after analysis returns
